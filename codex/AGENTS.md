# 默认代码风格

默认优先使用 **异常驱动的错误处理**，避免在每个步骤写冗余的参数防御式检查。
代码要求简洁、可维护。
如果目标语言、现有仓库风格、性能约束或接口契约明显不适合异常模型，则以项目约定为准。
C++ 代码优先考虑性能，但不要为了微小收益牺牲正确性和可维护性。

# 先规划后执行

对于复杂任务，先给出步骤化计划。

如果任务会修改多个文件、改变现有行为、改变接口或数据格式、安装依赖、联网下载内容、运行较重命令，先给计划并等待确认。

如果任务是只读型分析，例如阅读仓库、解释算法、梳理执行流、定位算法核心，可以先给简短计划，然后直接完成第一阶段分析，不必等待确认。

简单问题、局部小改动、纯解释性回答，可直接执。

# 阅读算法仓库的默认偏好

当目标是理解算法仓库时，默认优先：
1. 找到真实入口
2. 先建立端到端主流程
3. 区分算法核心与工程胶水
4. 抽取算法骨架
5. 每次只深读一个小的核心路径

解释代码时按语义块讲，不要做整仓逐行翻译。
优先帮助我建立心智模型，而不是罗列细节。
对日志、CLI、配置、包装层、明显 plumbing 默认压缩处理，除非它们直接影响算法行为。
对设计评价要区分：代码证据、合理推断、推测。
不确定时明确说明。
